# codebase-wiki.md — proposed format

**Status:** proposal, pending eng-lead review. Not yet implemented as a PM-OS artifact.

**Purpose:** a single, human-shareable document that summarizes what a codebase does, how it's built, and where its rough edges are — grounded in file citations, not prose recall. Intended to be handed to an engineering lead (or any new contributor) for a quick, structured review rather than a freeform code tour.

**Provenance:** this format adapts the existing `00c` codebase-understanding artifact that PM-OS's `/pm-context-import` already produces internally (see [`pm-context-scan-codebase/SKILL.md`](../../skills/pm-context-scan-codebase/SKILL.md)). The section spine is unchanged; this version adds confidence tiers and an open-questions tail so an external reviewer can tell what's verified fact vs. inference, and flag gaps directly.

**Hard rule:** every claim in every section must cite a file path (`see <path>`, or a line range for precision). No unsourced assertions — if something can't be traced to a file, it goes in "Open questions," not stated as fact.

---

## 1. TL;DR

One paragraph: what the product does, who it's for, the core tech stack, and a signal of scale/maturity (prototype vs. production, team size if evident, age of codebase).

> *Why it's here:* gives the reviewer a 10-second orientation before they read anything else. If this paragraph is wrong, everything downstream is probably wrong too — react to this first.

## 2. Current features & flows

User-facing capabilities, each tied to an entry-point file path.
`- Feature: <description> (see <path>)`

> *Why it's here:* confirms the scan actually found the product's real surface area, not just its scaffolding. An eng lead can skim this list and immediately say "you missed X" or "Y is dead code, not a live feature."

## 3. Architecture & modules

Module/package structure, optionally a diagram or table, each module cited to its path.

> *Why it's here:* this is the section most likely to carry tribal knowledge the scan can't infer (why a module is split the way it is, historical reasons for an odd boundary). Flag it as the one to read most carefully.

## 4. Data model

Key entities, relationships, and the storage layer, each citing where it's defined.

> *Why it's here:* wrong data-model assumptions break everything built on top of this doc later (specs, TRDs, onboarding). Worth an explicit "does this match reality" check.

## 5. Tech stack & dependencies

Languages, frameworks, key libraries, runtime, infra — with versions where known.

> *Why it's here:* fast to verify, fast to get stale. Good candidate for a lightweight periodic refresh rather than deep review each time.

## 6. Design language

Design system, token files, component library, or UI framework — explicitly noted as "not found" if absent, never silently skipped.

> *Why it's here:* absence is itself a finding (e.g. "no design system" tells a reviewer something about maturity/consistency risk). Don't let this section vanish just because there's nothing to report.

## 7. Integration points

External APIs, third-party services, auth, queues, webhooks, each with a usage citation.

> *Why it's here:* the highest-risk section to get wrong — missing an integration here means someone downstream doesn't know a dependency exists. Ask the eng lead to specifically confirm completeness.

## 8. Known constraints & tech debt

TODOs, FIXMEs, deprecated patterns, duplicated modules.

> *Why it's here:* this is where the scan's read of the code and the team's lived experience are most likely to diverge — the eng lead may know debt that isn't marked in-code at all. Treat gaps here as expected, not a scan failure.

---

## Cross-cutting conventions

**Confidence tiers** — each section carries a one-line marker:
`> Confidence: High / Medium / Low`
High = directly read from code with no interpretation. Medium = inferred from patterns (e.g. "this looks like the auth layer based on naming/imports"). Low = thin evidence, worth explicit confirmation.

**Open questions & uncertainties** (closing section) — anything the scan couldn't resolve: ambiguous ownership, contradictory patterns, missing pieces. This is the section built specifically for eng-lead comments — treat it as a review checklist, not a footnote.

---

## What's deliberately left out (vs. the internal `00c`/`00w` templates)

- **Stage-affinity tags** (`<!-- stage-affinity: 01 02 -->`) — meaningful to PM-OS's own pipeline routing, meaningless to an external reviewer.
- **`src_NNN` source IDs** — PM-OS's provenance system for its own evidence ledger; a file-path citation serves the same purpose here without the extra indirection.

If this format is approved, the natural next step is deciding whether it's generated by the existing `pm-context-scan-codebase` skill with light output changes, or whether it needs its own skill/script — that's a separate decision once the content shape itself is signed off.
